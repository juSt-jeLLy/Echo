# Requirements Document

## Introduction

This feature replaces the current "generate all audio upfront" approach in Documentary mode with a **progressive/streaming loading strategy**. Instead of synthesising all 4 TTS segments before playback begins, the system generates only the first segment's audio at start time, then prefetches each subsequent segment in the background as the user approaches the end of the current one. This reduces initial loading time and avoids spending ElevenLabs API credits on segments the user never reaches.

The Groq script generation (cheap text) still produces all 4 segments upfront. Only ElevenLabs TTS calls are deferred.

**Playback structure:** max 4 parts × ~40 seconds each = ~2 min 40 sec total documentary.

## Glossary

- **App**: The City Whispers single-page application.
- **DocumentaryService**: The service in `src/services/documentaryService.ts` responsible for Groq script generation and ElevenLabs TTS synthesis.
- **DocumentaryScript**: A 4-segment plain-text script generated by Groq, each segment ~85 words (~40 seconds at ~130 wpm).
- **Segment**: One of the 4 parts of a DocumentaryScript (~85 words, ~40 seconds of audio).
- **SegmentIndex**: A zero-based integer (0–3) identifying a Segment's position in the DocumentaryScript.
- **TTS**: Text-to-speech synthesis via ElevenLabs API.
- **AudioURL**: An object URL pointing to a synthesised audio blob for a single Segment.
- **SegmentCache**: An in-memory map from SegmentIndex to AudioURL, populated progressively during playback.
- **PrefetchTrigger**: The playback time threshold (30 seconds into the current Segment) at which the next Segment's TTS synthesis begins in the background.
- **useDocumentary**: The React hook in `src/hooks/useDocumentary.ts` managing documentary playback state.
- **NarratorVoice**: One of the 6 fixed ElevenLabs voices available for DocumentaryMode narration.
- **AbortController**: A browser API used to cancel in-flight fetch/TTS requests when the user stops or navigates away.
- **PlaybackTime**: The `currentTime` property of the active `HTMLAudioElement`, measured in seconds.

## Requirements

### Requirement 1: Segment Word Count and Duration

**User Story:** As a product owner, I want each documentary segment to be approximately 85 words so that each segment produces roughly 40 seconds of audio, keeping the total documentary under 3 minutes.

#### Acceptance Criteria

1. THE DocumentaryService SHALL instruct Groq to generate each Segment with a target of 80–90 words.
2. THE DocumentaryScript SHALL consist of exactly 4 Segments separated by `---`.
3. IF Groq returns fewer than 2 Segments, THEN THE DocumentaryService SHALL throw a descriptive error message.
4. THE DocumentaryService SHALL pad the Segment array to length 4 by repeating the last Segment if Groq returns 2 or 3 Segments.

---

### Requirement 2: Immediate First-Segment Playback

**User Story:** As a user, I want audio to start playing as soon as the first segment is ready, so that I don't have to wait for all four segments to be synthesised before I can listen.

#### Acceptance Criteria

1. WHEN DocumentaryMode audio generation begins, THE useDocumentary hook SHALL synthesise only the first Segment (SegmentIndex 0) via TTS before starting playback.
2. WHEN the first Segment's AudioURL is ready, THE useDocumentary hook SHALL begin playback immediately without waiting for remaining Segments.
3. THE useDocumentary hook SHALL set `isLoading` to `false` and `isPlaying` to `true` as soon as the first Segment begins playing.
4. THE useDocumentary hook SHALL NOT call TTS for Segments 1, 2, or 3 before playback of Segment 0 has started.

---

### Requirement 3: Background Prefetch at 30-Second Mark

**User Story:** As a user, I want the next segment to be ready before the current one ends, so that playback continues seamlessly without gaps.

#### Acceptance Criteria

1. WHEN PlaybackTime of the currently playing Segment reaches 30 seconds, THE useDocumentary hook SHALL begin synthesising the next Segment's TTS in the background.
2. THE prefetch SHALL occur for Segments 0 → 1, 1 → 2, and 2 → 3 only; WHEN Segment 3 (the last) is playing, THE useDocumentary hook SHALL NOT initiate any further TTS synthesis.
3. THE prefetch SHALL be initiated at most once per Segment transition; THE useDocumentary hook SHALL NOT re-trigger prefetch if the 30-second threshold is crossed multiple times (e.g. due to seek or timer re-evaluation).
4. WHILE a prefetch is in progress, THE useDocumentary hook SHALL continue playing the current Segment uninterrupted.

---

### Requirement 4: No Prefetch if User Stops Before 30 Seconds

**User Story:** As a product owner, I want TTS API calls to be skipped for segments the user never reaches, so that we avoid unnecessary ElevenLabs API costs.

#### Acceptance Criteria

1. IF the user pauses playback before PlaybackTime reaches 30 seconds on the current Segment, THEN THE useDocumentary hook SHALL NOT trigger prefetch for the next Segment.
2. IF the user resumes playback after having paused, AND PlaybackTime is at or beyond 30 seconds at the time of resumption, THEN THE useDocumentary hook SHALL trigger prefetch for the next Segment.
3. IF the user stops the documentary entirely (navigates away or closes the panel) before PlaybackTime reaches 30 seconds on any Segment, THEN THE useDocumentary hook SHALL abort all in-flight TTS requests via AbortController.

---

### Requirement 5: Seamless Sequential Playback

**User Story:** As a user, I want each segment to play immediately after the previous one ends, so that the documentary feels like a continuous audio experience.

#### Acceptance Criteria

1. WHEN a Segment ends, THE useDocumentary hook SHALL immediately begin playing the next Segment if its AudioURL is already in the SegmentCache.
2. IF the next Segment's AudioURL is not yet in the SegmentCache when the current Segment ends, THEN THE useDocumentary hook SHALL set `isLoading` to `true` and wait until the AudioURL is available before playing.
3. WHEN the last Segment (SegmentIndex 3) ends, THE useDocumentary hook SHALL set `isPlaying` to `false` and `isLoading` to `false`.
4. THE useDocumentary hook SHALL play Segments in ascending SegmentIndex order (0, 1, 2, 3) without skipping or repeating.

---

### Requirement 6: Single-Segment TTS Function

**User Story:** As a developer, I want a dedicated function to synthesise a single Segment, so that the progressive loading logic can call TTS on demand for individual segments.

#### Acceptance Criteria

1. THE DocumentaryService SHALL expose a `synthesiseSegment(segment: string, voiceId: string, signal?: AbortSignal): Promise<string>` function that synthesises a single Segment and returns one AudioURL.
2. WHEN `synthesiseSegment` is called with a valid segment and voice, THE DocumentaryService SHALL return an AudioURL within a reasonable time.
3. IF the ElevenLabs TTS call fails with a 429 rate-limit error, THEN THE DocumentaryService SHALL retry with exponential backoff up to 3 times before throwing.
4. IF `synthesiseSegment` is called with an aborted signal, THEN THE DocumentaryService SHALL throw an `AbortError` immediately without making a network request.
5. THE existing `synthesiseDocumentaryAudio` function SHALL be updated to delegate to `synthesiseSegment` internally, so that retry and voice settings logic is not duplicated.

---

### Requirement 7: Playback Time Tracking

**User Story:** As a developer, I want the hook to monitor the audio element's current playback time, so that the 30-second prefetch trigger can fire at the correct moment.

#### Acceptance Criteria

1. THE useDocumentary hook SHALL attach a `timeupdate` event listener to the active `HTMLAudioElement` to monitor PlaybackTime.
2. WHEN the `timeupdate` event fires and PlaybackTime is ≥ 30 seconds, THE useDocumentary hook SHALL evaluate whether a prefetch should be triggered for the next Segment.
3. THE useDocumentary hook SHALL remove the `timeupdate` event listener when the component unmounts or when the documentary is reset.
4. THE useDocumentary hook SHALL track which Segments have already had prefetch triggered, using a ref, to prevent duplicate TTS calls.

---

### Requirement 8: Abort and Cleanup

**User Story:** As a developer, I want all in-flight TTS requests to be cancelled when the user leaves or resets the documentary, so that there are no memory leaks or orphaned API calls.

#### Acceptance Criteria

1. WHEN the city, era, or voice inputs to useDocumentary change, THE useDocumentary hook SHALL abort all in-flight TTS requests and revoke all existing AudioURLs from the SegmentCache.
2. WHEN the useDocumentary hook unmounts, THE useDocumentary hook SHALL abort all in-flight TTS requests and revoke all AudioURLs.
3. THE useDocumentary hook SHALL use a single AbortController per generation session, shared across all TTS calls within that session.
4. WHEN `retry()` is called, THE useDocumentary hook SHALL abort the current session, clear the SegmentCache, and restart from Segment 0.
